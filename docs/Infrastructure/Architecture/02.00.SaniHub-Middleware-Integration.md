## 1. Contexte et gouvernance

### 1.1 Résumé exécutif

**SaniHub** est un middleware d'intégration qui centralise et structure les flux de données inter-applicatifs chez Sani Marc. Actuellement, les applications (Infor M3, TrueCommerce, Dynamics 365, Ortems, BBX) communiquent via une base de données de transition (MariaDB) ou des intégrations point-à-point fragiles. SaniHub transforme ce modèle en une plateforme d'orchestration d'intégration unique, offrant traçabilité complète, gestion des erreurs robuste et scalabilité pour supporter la croissance.

**Valeur métier attendue :**
- Réduire le temps de diagnostic d'une commande manquée de 4h à 15 min (traçabilité)
- Éliminer les pertes de données dues aux défaillances d'intégration (fiabilité)
- Accélérer le déploiement de nouvelles intégrations (de 3 semaines à 3 jours)
- Réduire les coûts d'opération de transition MariaDB (migrations vers SaniHub phases après)

**Budget estimé :** ~600k$ (infrastructure + développement + déploiement)  
**Timeline :** MVP Q2 2026, Go-Live Q4 2026  
**Approche :** Phases parallèles (tous les flux majeurs dès MVP)

### 1.2 Objectifs métier

| ID | Objectif | Métrique de succès | Délai | Impact |
|----|---------|--------------------|-------|--------|
| OBJ-01 | Centraliser et tracer tous les flux inter-applicatifs | 100% des flux documentés et tracés dans SaniHub | Q4 2026 | Diagnostique d'incident en 15 min (vs 4h) |
| OBJ-02 | Réduire les défaillances d'intégration | 99.95% succès de transmission de messages | Q4 2026 | 0 perte de commande, 0 duplication |
| OBJ-03 | Accélérer déploiement nouvelles intégrations | Nouvelle intégration en 3 jours (vs 3 semaines) | Q4 2026 | Vélocité +10x |
| OBJ-04 | Préparer migration de MariaDB de transition | 80% des usages MariaDB remplacés par SaniHub | Q1 2027 | Dés-commission MariaDB |

### 1.3 Piliers métier (Business Drivers)

1. **Maîtrise des flux inter-applicatifs** : Aujourd'hui fragmentés, difficiles à auditer et à debugger. SaniHub offre un point d'observation centralisé.
2. **Fiabilité et intégrité des données** : Les intégrations d'aujourd'hui ne garantissent pas la livraison. SaniHub ajoute des garanties (idempotence, replay, dead-letter queues).
3. **Conformité opérationnelle** : Auditabilité complète pour respecter les normes de traçabilité et conformité réglementaire.
4. **Efficacité opérationnelle** : ReqReduire l'overhead de maintenance des intégrations ad hoc, gagner en vélocité de développement.

### 1.4 Périmètre

#### Inclus (MVP + Go-Live)

- **Flux de commandes clients** (Order-to-Cash)
  - Création de commande : Dynamics 365 → M3 → Ortems
  - Confirmation/statut d'expédition : M3/Ortems → TrueCommerce → Dynamics 365
  
- **Flux EDI/B2B** (TrueCommerce)
  - Reçus de commandes clients (850 EDI)
  - Confirmations et avis d'expédition (856, 857 EDI)
  - Intégration avec middleware EDI existant

- **Flux de production** (Procure-to-Pay, Plan-to-Produce)
  - Commandes d'achat : M3 → Fournisseurs (fichiers/API)
  - Réceptions et facturations inversées
  - Plans de production M3 → Ortems → BBX

- **Flux internes** (Inventory, Stock)
  - Mouvements de stock : M3 → BBX (entrepôts)
  - Synchronisation de masters (articles, clients, fournisseurs)

- **Connecteurs** pour :
  - Infor M3 (OData API / SQL)
  - Dynamics 365 (REST API / Webhooks)
  - TrueCommerce (API REST + EDI)
  - Ortems (API + fichiers)
  - BBX (API locale + appels procédures stockées)
  - MariaDB transitoire (migration progressive)

- **Infrastructure** :
  - On-premise (Sani Marc maîtrise l'hébergement)
  - Déploiement Kubernetes (containerisé)
  - Base de données logs (PostgreSQL ou SQL Server)
  - Message broker (RabbitMQ ou Kafka)

#### Exclu (Phases futures 02)

- **Temps réel ultra-basse latence** : MVP cible latence <2s (acceptable pour commandes), phase ultérieure pour critères <500ms
- **Orchestration de workflows métier complexes** : BPMN/workflows au-delà de routage simple (future phase)
- **Analytics/data warehouse** : BI / requêtes analytiques (considérer pipeline CDC séparé futurementôt)
- **Master data management (MDM)** : Harmonisation des données (phase ultérieure avec partenaire spécialisé)

### 1.5 Alignement stratégique d'entreprise

**SaniHub** s'inscrit dans la **Phase 1 de la Stratégie d'Intégration de Sani Marc (2026)**, qui vise à moderniser l'architecture d'intégration et préparer la migration cloud à moyen terme. Il contribue aux capacités d'entreprise :
- **Intégration applicative** : Hub centralisé
- **Gouvernance de données** : Traçabilité et conformité
- **Agilité opérationnelle** : Déploiement rapide d'intégrations

**Principes architecturaux respectés :**
- Standardisation (événements canoniques, contrats d'API)
- Traçabilité (logging centralisé, corrélation de requêtes)
- Résilience (idempotence, retry policies, DLQ)
- Sécurité (authentification/ autorisation par connecteur, chiffrement in-transit)

## 2. Vision architecturale

### 2.1 La problématique "as-is"

```
Aujourd'hui (fragmentation) :

Dynamics 365 ──┐
Infor M3 ──────┼──→ MariaDB (transition) ──→ Ortems
TrueCommerce ─┤
BBX ───────────┼──→ Scripts/Fichers/APIs ad-hoc
               └──→ Intégrations point-to-point directes (E2E)

Symptômes :
- "Où est passé la commande 12345 ?" → 4h de debug en multi-équipes
- Duplication de logique : batch jobs + APIs + scripts mixés
- Défaillances non visibles : message perdu = pas d'alerte
- Coût opérationnel élevé : maintenance 15-20% effort IT
```

### 2.2 La vision "to-be"

```
Demain (centralisation) :

Dynamics 365       TrueCommerce       MariaDB        BBX
    ↓                  ↓                 ↓             ↓
    └──────────────────┴─────────────────┴─────────────┘
                         ↓
                    ┌─────────────┐
                    │  SaniHub    │ (Message Router + Orchestration)
                    │   Hub d'int │                    │
                    │ Converters  │                    │
                    └─────────────┘                    │
                    ↓    ↓    ↓    ↓                  │
         Infor M3  Ortems  Warehouse  Fournisseurs   │
                              
Features :
✓ Routage intelligent (saga policies, failover)
✓ Traçabilité universelle ("Commande 12345 : reçue → envoyée M3 → reçue → envoyée Ortems")
✓ Idempotence garantie (pas de duplication)
✓ Dead-letter queues (capture les messages non-routables)
✓ Retry policies (2x avec backoff exponentiel)
✓ Monitoring + Alertes (SLA par flux)
✓ API de replay manuel (retraitement après fix production)
```

### 2.3 Principes de conception

| Principe | Définition | Application SaniHub |
|----------|------------|-------------------|
| **Centralisation** | Un seul point de passage pour tous les flux inter-app | Hub unique, tous les connecteurs → SaniHub |
| **Transparence** | Chaque événement tracé avec ID de corrélation | Logging centralisé + Dashboard (Kibana/Grafana) |
| **Idempotence** | Aucun impact si un message est traité 2x | Deduplication key + storage des message IDs |
| **Resilience** | Gestion gracieuse des défaillances | Retry, exponential backoff, DLQ, circuit breaker |
| **Modèle canonique** | Format intermédiaire commun (CommandeClient, ODP, etc) | Événements canoniques + transformateurs par connecteur |
| **API-first** | Toute interaction via API documentées | REST/gRPC, contrats OpenAPI |
| **Security** | Zero-trust, authentification par connecteur, chiffrement | OAuth2 interne, TLS, secrets management |

## 3. Architecture métier et processus

### 3.1 Processus métier clés impactés

#### 3.1.1 Order-to-Cash (Commande client)

```
Scénario main : Commande client B2B

1. Client émet commande (EDI 850) →:TradeCommerce reçoit
2. TrueCommerce poste event "CommandeClientRecue" → SaniHub
3. SaniHub centralise, valide, formate (canonical model) → event "CommandeClientCree"
4. M3 reçoit event (via connecteur), crée commande → event "CommandeM3Creee"
5. Ortems reçoit event (via connecteur), crée PDP → event "PDPCreee"  
6. Stock FIFO réservé
7. Picklist généré et déployé à BBX
8. Expédition confirmée → event "ExpeditionPreparee"
9. TrueCommerce reçoit event, envoie 856 (expédition) au client
10. Dynamics 365 reçoit event, marque commande comme "Expediee"

Traçabilité SaniHub : Tous les events pour commande 12345 dans le catalogue central
```

#### 3.1.2 Procure-to-Pay (Commande achat)

```
Scénario : Commande achat auprès fournisseur externe

1. Planification M3 génère besoin → event "BesoinsCreationDemandee"
2. SaniHub reçoit, enrichit avec master fournisseur
3. Crée commande achat → event "CommandeAchatCreee"
4. Envoie au fournisseur (EDI 850 ou email/API fournisseur)
5. Réception marchandise → event "ReceptionMarchandiseConfirmee"
6. M3 reçoit, met à jour disponibilité → event "StockMisAJour"
7. Facture fournisseur reçue → event "FactureRecue"
8. Dynamics 365 reçoit pour reconciliation comptable

Traçabilité SaniHub : Tous les events pour bon de commande achat XXX
```

#### 3.1.3 Plan-to-Produce (Production)

```
Scénario : Plan de production

1. Planification M3 génère un ordre de fabrication → event "OdfPlanifie"
2. SaniHub reçoit, traduit (format Ortems)
3. Envoie à Ortems → ODF reçue et confirmée
4. Ortems planifie, réserve ressources → event "OdfAssigneeRessources"
5. BBX reçoit tasks (via interface actuelle à moderniser)
6. Picklist à préparation → event "PicklistGeneree"
7. Livraison matière première → event "MatierePremiereLivree"
8. Travail en cours → statut d'avancement
9. Produit fini → event "ProduitFiniGenere"
10. Stock M3 mis à jour → event "StockCompleteProduitFini"

Traçabilité SaniHub : Tous les events pour ODF XXX (de planification à livraison)
```

### 3.3 Modèle de données canonique (haut niveau)

SaniHub définit un ensemble d'événements/modèles métier **canoniques** que tous les connecteurs comprennent :

| Entité | Champs clés | Source dominante | Utilisateurs |
|--------|------------|------------------|-------------|
| **CommandeClient** | ID, dateCreation, client, lignes[], adresse_livraison, statut | Dynamics 365 / TrueCommerce | M3, Ortems, TrueCommerce |
| **CommandeAchat** | ID, fournisseur, lignes[], dateLivraison, statut | M3 | Fournisseurs (EDI/API) |
| **OrdreFabrication** | ID, dateDebut, dateFin, produit[], ressources[], statut | M3 | Ortems, BBX |
| **Expédition** | ID, commande_id, adresse, date_préparée, date_expédiée, numéro_tracking | M3 / Ortems | TrueCommerce, Dynamics 365 |
| **Réception** | ID, bon_réception, commande_achat_id, quantités[], statut | M3 | Comptabilité (Dynamics 365) |
| **Facture** | ID, fournisseur, montant, devise, statut_paiement | Dynamics 365 / Fournisseur | Comptabilité, Trésorerie |
| **StockMovement** | ID, article, quantité, magasin, type (entrée/sortie) | M3 | BBX, Dynamics 365 (reporting) |

Chaque événement porte un **correlation_id** unifié (ex : commande_id) pour traçabilité.

## 4. Architecture de solution

### 4.1 Style architectural retenu : Event-Driven Integration Hub

**Justification :**
- Découplage temporal : Producteur n'attend pas consommateurs (async)
- Scalabilité : Ajouter connecteur sans modifier hub
- Résilience : Messages en queue → replay après incident
- Traçabilité : Event log = source de vérité (immutable)
- Modèle familier : Pattern ESB/iPaaS largement adopté

**Patterns appliqués :**
- **API Gateway** : Point d'entrée unique, authentification centralisée
- **Message Broker** : Découplage événements (RabbitMQ ou Kafka)
- **Saga Pattern** : Orchestration de flux multi-étapes (Order-to-Cash)
- **Dead-Letter Queue** : Capture messages non routables
- **Event Sourcing** : Audit log immutable de tous les events
- **CQRS** : Modèle d'écriture (events) ≠ modèles de lecture (projections pour chaque app)

### 4.2 Vue d'ensemble C4 — Contexte (Niveau 1)

```
External Actors:
  - Clients B2B (via EDI)
  
Internal Systems:
  [Dynamics 365]  [Infor M3]  [TrueCommerce]  [Ortems]  [BBX]  [MariaDB]
  
         ↓              ↓            ↓          ↓        ↓        ↓
         
         └──────────────────────────────────────────────────────┘
                              ↓
                        ┌────────────┐
                        │  SaniHub   │
                        │   Hub d'int│
                        │(Middleware)│
                        └────────────┘
                              ↓
         ┌──────────────────────────────────────────────────────┐
         ↓              ↓            ↓          ↓        ↓        ↓
         
  [Dynamics 365]  [Infor M3]  [TrueCommerce]  [Ortems]  [BBX]  [Fournisseurs]

Actors humains :
  - Équipe IT (monitoring, support)
  - Équipe métier (ventes, production, achats → via applications)
```

**Description :** SaniHub reçoit des messages d'applications métier et d'acteurs externes (clients B2B), les centralise, transforme et route vers destinations. Agit comme point unique d'intégration pour tous les flux.

2. **Router & Orchestrator** : Cœur logique, décide destinataires (saga orchestration)
3. **Event Transformer & Validation** : Convertit canonical → format cible, valide schémas
4. **Message Broker** (RabbitMQ ou Kafka) : Queue asynchrone pour découplage
5. **Connector Manager & Registry** : Enregistre et maîtrise connecteurs (adapters pour M3, Dynamics, Ortems, BBX, TrueCommerce)
6. **Event Store (PostgreSQL)** : Log immuable de tous les events (source de vérité)
7. **State Store** (Redis ou Bolt) : Cache de state transitoire (en-cours transactions, dedup keys)
8. **Observability Stack** : Prometheus (metrics) + Logs (ELK/Datadog) + Traces (Jaeger)
9. **Dashboard & Control Plane** : UI pour monitor, replay messages, configurer routage

| **Dynamics 365** | REST API + Webhooks | Graphe | Bidirectionnel | Quasi temps réel | <2s, 99.5% uptime |
| **TrueCommerce** | REST API + EDI | HTTP/SFTP | Bidirectionnel | Batch 1h | <5min, 99% uptime |

### 5.2 Contrats d'API SaniHub

#### API Produit (pour applications métier)

```
/api/v1/events/publish (POST)
  Authentification: OAuth 2.0 (client credentials)
  Body: {
    "event_type": "CommandeClientCreee",  // Événement canonique
    "correlation_id": "CMD-12345",         // ID métier unique
```

## 8. Sécurité et conformité

### 8.1 Modèle de menaces STRIDE

| Menace | Description | Mitigation | Risque résiduel |
|--------|-------------|-----------|-----------------|
| **Spoofing (Authentification)** | App usurpe identité app autorisée | OAuth 2.0 client credentials + TLS mutual | Très faible |
| **Tampering (Intégrité)** | Attaquant modifie message en transit | TLS encryption + HMAC signature interne | Très faible |
| **Repudiation** | App nie avoir envoyé message | Audit log immuable (EventStore), corrélation ID | Très faible |
| **Information Disclosure** | Fuite données sensibles (commandes, prix) | Chiffrement au repos (AES-256), masquage logs | Faible |
| **Denial of Service (DoS)** | Attaquant inonde SaniHub de requêtes | Rate limiting API Gateway (par app), circuit breaker | Moyen (mitigé) | 
| **Elevation of Privilege** | Process accède ressource non autorisée | RBAC (Service Accounts), pod security policies | Très faible |

### 8.2 Stratégie d'authentification & autorisation

```
Autentification :
  - Input API Gateway ← App métier : OAuth 2.0 client credentials 
    (M3_user/@sanimark.com, client_id + client_secret via HashiCorp Vault)
  - Webhook entrante ← App externe (Ortems, TrueCommerce) : 
    OAuth 2.0 bearer token (JWT signé, rotation 90j)

Autorisation (RBAC) :
  - M3 can publish: CommandeAchat*, ODF*, StockMovement*
  - Dynamics 365 can publish: CommandeClient*, Facture*, Expédition*
  - TrueCommerce can publish/subscribe: CommandeClient*, Expédition*
  - Ortems can subscribe: ODF*, can publish: ODF.Confirme, ODF.Avancement
  - BBX can subscribe: Picklist*, can publish: PicklistComplete, ExpeditionPreparee
  - Admin (Mathieu + Paul) : full access + replay
  
ACL par event_type (whitelist), implémenté au router
```

### 8.3 Stratégie de chiffrement

```
En transit (TLS 1.3) :
  - API Gateway → Infor M3 OData API (point authentication mutual)
  - API Gateway → Dynamics 365 REST API
  - Connecteur → TrueCommerce API
  - Connecteur → Ortems API
  - Connecteur → BBX local API
  
Au repos (Encryption at Rest) :
  - PostgreSQL EventStore : AES-256 (transparent via dm-crypt kernel module)
  - Redis StateStore : Redis ACL + TLS (optional Redis encryption)
  - Secrets (DB passwords, API keys) : HashiCorp Vault (encrypted + rotation policy)
  - Logs archivés (NFS) : GPG encrypted, keyfile in separate secure location
  
Gestion des secrets :
  - Vault centralisé (Sani Marc IT)
  - Injection secrets in Pod via Secret engine
  - Rotation automatique 90 jours (M3, Dynamics tokens)
  - Audit de chaque accès secret (Vault log)
```

### 8.4 Conformité

- **PIPEDA (Canada)** : Données clients critiques, droit d'accès/suppression → audit log SaniHub justifie conformité
- **Audit trail** : 24 mois d'événements immuables, identité source + target + action

## 10. Décisions architecturales (ADRs)

### ADR-001 : Event-Driven Architecture (vs Point-to-Point)

**Statut :** ✅ Retenue

**Contexte :** Sani Marc a 50+ intégrations adhoc (scripts, fichiers, APIs directes). Fragilité et coût de maintenance élevé. Besoin : centralisé, tracer, résilience.

**Options considérées :**
1. **Event-Driven (Acceptée)** : Message broker central, événements canoniques, discovery connecteurs
2. **Point-to-Point amélioré** : API gateway + routeur simplifié (pas asynchrone)
3. **Hybrid** : Synchrone pour commandes critiques, async pour stock

**Justification :** Event-driven impose une seule fois, découplage à long terme, scalable, audit friendly.

**Conséquences positives :**
- Découplé, facile à tester connecteurs
- Replay naturel en cas d'erreur destination

**Conséquences négatives :**
- Eventual consistency (commande apparaît progressivement dans Ortems)
- Complexité debugging (asynchrone)

**Risques & mitigations :** 
- Retard détection erreur (peut prendre 5-10 sec) → mitigé via DLQ et alertes
- Saga partial failures (certains connecteurs traient avant failure) → mitigé via saga pattern + compensating transactions

---

### ADR-002 : Plateforme On-Premise vs Cloud

**Statut :** ✅ Retenue (On-Premise)

**Contexte :** Sani Marc exploite déjà infrastructure on-prem (M3, spas, BBX). Cloud = coûts additionnels (licensing Dynamics 365 + data egress) + compliance.

**Options :**
1. **On-Premise (Retenue)** : Kubernetes on company servers
2. **Azure Hybrid** : Core SaniHub in Azure, connecteurs on-prem
3. **Full Cloud (Azure)** : Entièrement Azure

**Justification :** Coût TCO minimal (CIO priority), latency < 2ms (same datacenter), security (no egress).

**Conséquences :**
- Gestion IT overhead (K8s, PaaS, upgrades)
- Scaling vertical < horizontal (limited hardware)

**Risques :**
- DR (backup at remote datacenter) → mitigated via daily snapshots NFS

---

### ADR-003 : PostgreSQL pour Event Store vs Event Sourcing Database

**Statut :** ✅ Retenue (PostgreSQL)

**Contexte :** SaniHub needs immutable event log (compliance, replay). Options : traditional RDBMS + append-only table vs specialized (EventStoreDB, Apache Kafka).

**Options :**
1. **PostgreSQL + JSONB (Retenue)** : Append-only table, JSONB for schema flexibility, fast Queries
2. **Event Sourcing DB** (EventStore Pro) : Optimized for events, built-in snapshots
3. **Kafka** : Streaming DB, excellent for CDC, partition by key, compaction

**Justification :** PostgreSQL : already known Sani Marc DBAs, JSONB supports evolution, SQL queries for compliance queries, cost-effective.

**Trade-off :** Kafka would be architectural purer but operational overhead higher.

---

### ADR-004 : Synchrone vs Asynchrone par flux

**Statut :** ✅ Retenue (Hybrid)

**Contexte :** Commande client = latency sensitive (customer waits), production = tolerance to 5-10 sec delay

**Décision :**
- **Synchrone** (gRPC request-response) : CommandeClient creation (Dynamics → M3), ExpéditionConfirmation
- **Asynchrone** (events via broker) : ODF routing, stock movements, historical batch jobs

**Justification :** Latency budget réaliste, scalabilité on async, resilience on async.

---

### ADR-005 : Replay Request API (Admin)

**Statut :** ✅ Retenue

**Contexte :** When Ortems is temporarily unavailable (30 min), commands queue. When Ortems comes back, replay needed. Manual vs automated?

**Options :**
1. **Manual Replay API** (Retenue) : Admin calls /replay/{correlation_id}, approves before re-sending
2. **Automatic Replay** : Policy-based (retry N times, then auto-replay on recovery)

**Justification :** Control, reduces risk of replay storms, audit trail (who replayed when).

**Risk :** Operator fatigue (manual replay 1000s of events) → mitigated via UI bulk replay UI.

## 11. Glossaire

| Terme | Définition | Contexte |
|-------|-----------|---------|
| **SaniHub** | Middleware central d'intégratio inter-applicative de Sani Marc | Projet |
| **Événement canonique** | Format d'événement intermédiaire standardisé (CommandeClient, ODF, etc) compris par tous connecteurs | Architecture |
| **Correlation ID** | Identifiant unique métier (ex : numéro commande CMD-12345) reliant tous les events d'une transaction | Traçabilité |
| **Saga Pattern** | Orchestration d'une transaction distribuée sur N services via compensating transactions en cas d'erreur | Pattern |
| **Dead-Letter Queue (DLQ)** | Queue spéciale capturant les messages qui ne peuvent pas être routés (connecteur down, schema mismatch) | Resilience |
| **Connecteur** | Adapter lisant/écrivant dans une application externe (M3, Dynamics, Ortems, BBX, TrueCommerce) | Architecture |
| **Message Broker** | Système de file d'attente (RabbitMQ, Kafka) découplant producteur et consommateur | Infrastructure |
| **Event Sourcing** | Patern stockant l'historique immuable des events plutôt que l'état final uniquement | Architecture |
| **Circuit Breaker** | Pattern protégeant un appel distant : open (fail-fast) → half-open (test) → closed (OK) | Resilience |
| **CQRS** | Command Query Responsibility Segregation : modèle d'écriture (events) distinct du modèle de lecture (projections) | Pattern |
| **Strangler Fig** | Pattern de migration progressive remplaçant graduellement ancien système par nouveau sans big-bang | Migration |
| **RPO** | Recovery Point Objective : perte de données tolérée (ex: 15 min = perte max 15 min events) | SLA |
| **RTO** | Recovery Time Objective : temps maximal à rétablir service après panne (ex: 5 min) | SLA |
| **TLS** | Transport Layer Security : protocole encryption communications en transit | Sécurité |
| **OAuth 2.0** | Standard d'authentification : app A demande jeton à autorité (Vault) avant appeler API | Sécurité |
| **Vault** | HashiCorp Vault : gestionnaire centralisé de secrets (passwords, API keys) | Infrastructure |
| **Latency p95/p99** | Percentile : 95% de requêtes < 2s (p95), 99% < 5s (p99) | Performance |
| **SLA** | Service Level Agreement : engagement de disponibilité (ex: 99.95% uptime) | SLA |
| **Throughput** | Nombre d'événements traités par seconde (ex: 1000 msg/sec) | Performance |
